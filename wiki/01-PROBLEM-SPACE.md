# 问题空间：我们要解决什么

> 核心问题：我们到底要解决什么问题？

## 个人 AI 助手 vs 企业 AI 助手

市面上的 AI 助手框架很多，但它们解决的是不同的问题。

### 企业级框架的特点

- **功能完整**：支持 RAG、知识图谱、多租户...
- **部署复杂**：需要 Kubernetes、向量数据库...
- **团队协作**：多用户、权限管理、审计日志...
- **成本高昂**：基础设施费用、维护成本...

代表项目：LangChain、LlamaIndex、AutoGPT...

### 个人 AI 助手的需求

作为个人用户，我们需要的是：

| 需求 | 说明 |
|------|------|
| **轻量** | 一台 VPS 就能跑，不依赖复杂基础设施 |
| **可控** | 自己的代码、自己的数据、自己的模型选择 |
| **灵活** | 想用什么模型就用什么，想加什么功能就加 |
| **简单** | 几千行核心代码，读得懂、改得动 |

**nanobot 的定位**：一个超轻量级的个人 AI 助手框架，核心代码约 4000 行。

## 多通道接入的必要性

为什么需要支持多个聊天平台？

### 场景一：不同场景用不同平台

- 日常聊天用 **Telegram**
- 工作沟通用 **Slack**
- 游戏社区用 **Discord**
- 国内用户用 **飞书** 或 **微信**

### 场景二：统一入口

用户希望有一个 AI 助手，能够在所有这些平台上使用，而不是为每个平台单独配置。

### 场景三：成本考虑

Telegram 机器人免费，但某些企业场景需要 Slack。支持多平台让用户有选择权。

**nanobot 的解决方案**：

```
                    ┌──────────────┐
Telegram ──────────▶│              │
Discord ───────────▶│   nanobot    │────────▶ LLM
Slack ─────────────▶│   (统一)     │
WhatsApp ──────────▶│              │
Feishu ────────────▶│              │
                    └──────────────┘
```

**源码位置**：`nanobot/channels/`

## 多模型支持的必要性

为什么不能只用一个模型？

### 场景一：成本优化

| 模型 | 成本 | 适用场景 |
|------|------|---------|
| Claude 3.5 Sonnet | 高 | 复杂推理 |
| GPT-4o | 高 | 复杂推理 |
| DeepSeek V3 | 低 | 日常对话 |
| Claude Haiku | 低 | 简单任务 |

用户希望根据任务复杂度选择模型。

### 场景二：可用性

- OpenAI 在某些地区不可用
- Anthropic 只支持某些国家
- 私有部署的模型需要内网访问

### 场景三：模型切换

今天用 Claude，明天可能想试试 DeepSeek。不想被单一供应商锁定。

**nanobot 的解决方案**：

```python
# 配置文件
{
  "model": "openrouter/auto",  # 智能路由
  "providers": {
    "openrouter": { "apiKey": "sk-or-..." },
    "anthropic": { "apiKey": "sk-ant-..." },
    "deepseek": { "apiKey": "sk-..." }
  }
}
```

**源码位置**：`nanobot/providers/registry.py`

## 为什么要自己造轮子？

### 理由一：学习

理解一个 AI 助手框架的内部运作，最好的方式是自己写一个。

### 理由二：定制

市面上的框架要么太重，要么不够灵活。想要：
- 自定义记忆系统
- 自定义工具
- 自定义上下文构建
- 完全控制代理行为

### 理由三：可控

- 知道每一行代码在做什么
- 不依赖上游的 breaking changes
- 可以随时 fork 和修改

### 理由四：轻量

核心代码 ~4000 行，一个人可以读完、理解、修改。

```bash
# 统计核心代码行数
bash core_agent_lines.sh
```

## 目标用户画像

### 开发者

- 想要一个可定制的 AI 助手
- 不想学习复杂的框架
- 希望代码简洁易懂

### 研究者

- 研究 LLM 代理行为
- 需要控制实验变量
- 希望快速迭代

### 极客

- 想要一个"懂自己"的 AI 助手
- 愿意投入时间配置和调优
- 享受 DIY 的乐趣

---

## 不在范围内的问题

为了保持轻量，nanobot **不解决**以下问题：

| 不做 | 原因 |
|------|------|
| 多租户 | 个人使用不需要 |
| 复杂 RAG | 增加复杂度 |
| 分布式部署 | 个人 VPS 足够 |
| 企业级安全审计 | 个人场景不需要 |
| 可视化界面 | CLI 和聊天平台足够 |

**设计哲学**：做一件事，做好它。

```
└─────────────────────────────────────────────────────────────┘
```
